# Aggregate Parallel Research Fragments

Used by `crispy-research` after it fans out internal `explore` sub-agents (one per area or repo) per `SUBAGENTS.md` §5.1. Each sub-agent writes a partial-research markdown fragment to a temp file; this skill merges them into the single canonical `research.md`.

## When to use

- Only after a researcher fan-out triggered by the threshold **areas ≥ 3 OR repos ≥ 2** (§5.1).
- Never as a substitute for a single-shot research run when the threshold isn't met.

## Inputs

- `fragments` — list of absolute paths to partial-research markdown fragments, one per area/repo. Example:
  - `C:\repos\crispy-plugin\crispy-docs\<feature>\.tmp\research.auth.md`
  - `C:\repos\crispy-plugin\crispy-docs\<feature>\.tmp\research.payments.md`
  - `C:\repos\crispy-plugin\crispy-docs\<feature>\.tmp\research.notifications.md`
- `target` — absolute path to the final `research.md`, e.g. `C:\repos\crispy-plugin\crispy-docs\<feature>\research.md`.
- `feature_name` — used in the document header.

## Process

### 1. Verify blindness preservation in fragments

Before merging, verify NO fragment contains the feature name (search the spec for the feature title and any obvious feature-derived tokens, then check that no fragment file references them). If a leak is found, do NOT silently merge — record it as a `findings[*].severity: high` entry with `description: "blindness leak detected in fragment"` and `location` set to the offending fragment path. Continue with the merge so the human reviewer can see the contaminated content, but flag it loudly.

The aggregator's output `research.md` MAY include the feature name in the header (the researcher post-merge sets that), but **fragments themselves MUST NOT** reference the feature name or feature folder path (`SUBAGENTS.md` §10).

### 2. Prepend the standard blind-research header

Mirror `skills/create-research/SKILL.md`. The header reaffirms blindness:

```markdown
# Codebase Research: {feature_name}

> Generated by blind research (no spec.md was read).
> Aggregated from {N} partial fragments via `aggregate-research`.
```

### 3. Merge under the standard headings

Always emit these sections in this order, even if some fragments contributed nothing to a given section:

1. Architecture Overview
2. Logic Flows
3. Data Models
4. Integration Points
5. Test Coverage
6. Technical Debt
7. Anti-Patterns
8. Key Files Reference

For each section:

- Concatenate the matching section from every fragment.
- **Dedupe overlapping findings**: if two fragments make the same statement about the same `file:line`, keep one copy. Use exact-citation match plus normalized prose comparison; prefer the longer/more-specific phrasing.
- **Preserve every file/line citation** from each fragment. Citations are the audit trail — never drop them during merge.
- Within each section, group entries by source area/repo with a small subheading (e.g. `### Area: auth`) when more than one fragment contributed.

### 4. Conflict handling

If two fragments **disagree on a fact** (e.g. `package.json` in `repo-a` pins `lodash@4.17.20`, in `repo-b` pins `lodash@4.17.21`; or two fragments describe the same flow differently):

- **Keep BOTH statements.** Do not pick a winner.
- Attribute each to its source area/repo inline:

  ```markdown
  - lodash version (auth area, `repo-a/package.json:14`): `4.17.20`
  - lodash version (payments area, `repo-b/package.json:22`): `4.17.21`
  ```

- Add a one-line `> Conflict:` admonition above the entry so reviewers can find it quickly.
- Track every conflict so it appears in the `crispy-result` summary (§3).

### 5. Cleanup

After the merged `research.md` is written successfully:

- Delete every path in `fragments`.
- Remove the `.tmp` directory if it is now empty.
- Do NOT delete fragments if the write failed — return `status: failed` and leave the fragments in place for retry (§8).

### 6. Emit `crispy-result`

End with the standard fenced block (`SUBAGENTS.md` §3):

```yaml
status: ok | partial | failed
agent: aggregate-research
artifact_path: C:\repos\crispy-plugin\crispy-docs\<feature>\research.md
summary: |
  Merged {N} fragments into research.md.
  Total findings: {count}. Conflicts surfaced: {conflict_count}.
findings: []                    # aggregator is not a reviewer; leave empty
next_actions:
  - Proceed to Intent phase.
metadata:
  fragment_count: {N}
  total_findings: {count}
  conflicts:
    - section: Integration Points
      fact: "lodash version"
      sources: ["auth/repo-a", "payments/repo-b"]
```

## Failure handling

Per `SUBAGENTS.md` §8:

- Missing fragment file → `status: partial`, `next_actions: [provide missing fragment X]`. Do not invent content.
- Write failure on the target → `status: failed`, leave fragments untouched, retry once.
- Malformed fragment (no recognizable section headings) → `status: partial`, surface the offending fragment path.

## Anti-patterns

- ❌ Silently picking one side of a conflict to "clean up" the doc.
- ❌ Dropping `file:line` citations because the prose felt redundant.
- ❌ Deleting fragments before the target write succeeds.
- ❌ Reading `spec.md` during merge — aggregation inherits the blindness rule (§5.1, §10).
